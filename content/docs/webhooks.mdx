Webhooks push Blueticks events to your server in near real time.

This guide assumes you have an API key ([Authentication](/docs/authentication))
and an SDK client set up ([Quickstart](/docs/quickstart)).

## Event types

- `message.queued`, `message.sending`, `message.delivered`, `message.failed`, `message.read`
- `session.connected`, `session.disconnected`
- `campaign.started`, `campaign.paused`, `campaign.resumed`, `campaign.completed`, `campaign.aborted`

The enum also accepts legacy `*_webhook` event names for inbound and group
activity — e.g. `new_message_received_webhook`, `message_reaction_webhook`,
`ack_changed_webhook`, `poll_vote_webhook`, `reply_to_my_message_webhook`,
and the `participant_*` / `group_*` membership events. See the
[Create webhook reference](/docs/api/webhooks/create-webhook) for the full
enum.

## Payload shape

Every delivery POSTs:

```json
{
  "id": "evt_01h7...",
  "type": "message.delivered",
  "created_at": "2026-04-23T16:42:00.000Z",
  "data": { /* same shape as the resource being reported */ }
}
```

Headers:

```
Content-Type: application/json
Blueticks-Webhook-Id: wh_...
```

If you want to confirm a request really came from Blueticks, register a URL
that includes a hard-to-guess token (for example a path segment or query
string only you know) and reject any POST that doesn't carry it.

## Retries

Non-2xx responses or timeouts trigger up to 8 retries over ~24 h with
exponential backoff: `1m → 5m → 15m → 1h → 2h → 6h → 12h → 24h`.
After 8 consecutive failures the webhook auto-disables — you'll need
to PATCH `status: "enabled"` to re-enable after fixing your endpoint.
