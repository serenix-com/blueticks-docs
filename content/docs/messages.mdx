Every send goes through `POST /v1/messages`. The body is a discriminated
union — set `type` to one of `text`, `media`, or `poll`. Optional fields
`send_at`, `from`, and `reply_to` work on every variant.

## Send a text message

import blueticks

## Send media

`type: "media"` accepts an HTTPS URL plus an optional `kind`
(`image` · `video` · `audio` · `document` · `sticker` · `voice`).
When `kind` is omitted, the engine auto-detects from the URL or the
`Content-Type` of the fetched asset. `caption` shows under image/video
in WhatsApp; `filename` is shown for documents.

```bash
curl -X POST https://api.blueticks.co/v1/messages \
  -H "Authorization: Bearer BLUETICKS_API_KEY" \
  -d '{
    "to": "+972501234567",
    "type": "media",
    "media": {
      "url": "https://cdn.example.com/receipt.pdf",
      "kind": "document",
      "filename": "receipt-A42.pdf",
      "caption": "Order #A-42"
    }
  }'
```

Voice notes are media with `kind: "voice"`:

```json
{ "to": "+972...", "type": "media", "media": { "url": "https://.../note.ogg", "kind": "voice" } }
```

`media.url` must be `https://` and resolve to a non-private host. We
fetch and forward it on your behalf.

## Send a poll

`options` must contain 2–12 entries. `allow_multiple` lets recipients
pick more than one (default `false`).

```bash
curl -X POST https://api.blueticks.co/v1/messages \
  -H "Authorization: Bearer BLUETICKS_API_KEY" \
  -d '{
    "to": "+972501234567",
    "type": "poll",
    "poll": {
      "question": "Pizza tonight?",
      "options": ["Yes", "No", "Maybe"],
      "allow_multiple": false
    }
  }'
```

## Schedule for later

`send_at` (ISO 8601 with offset, ≥ 10 s in the future, ≤ 365 d out)
works on every type. The response status is `scheduled` instead of
`queued`.

```json
{ "to": "+972...", "type": "text", "text": "Reminder", "send_at": "2026-12-01T09:00:00Z" }
```

## Quote-reply to a prior message

Set `reply_to` to the wire `key` of a message in the same chat
(returned in `MessageResponse.key` once the original has been sent).
Works with `text`, `media`, and `poll`.

```json
{ "to": "+972...", "type": "text", "text": "Got it 👍", "reply_to": "false_972...@c.us_3EB0..." }
```

## React to a message

Reactions are a separate endpoint — `POST /v1/chats/{chat_id}/messages/{key}/reactions`
with body `{ "emoji": "❤️" }`. Pass an empty string to remove a reaction.
Use the `key` from `MessageResponse`, not the `id`.

```bash
curl -X POST "https://api.blueticks.co/v1/chats/972...@c.us/messages/false_...@c.us_3EB0.../reactions" \
  -H "Authorization: Bearer BLUETICKS_API_KEY" \
  -d '{"emoji":"❤️"}'
```

## Idempotency

Safe retries via the `Idempotency-Key` header. Same key + same body
within 24 h returns the original response. Different body with the
same key returns `409 idempotency_key_conflict`.

```bash
curl -X POST https://api.blueticks.co/v1/messages \
  -H "Authorization: Bearer BLUETICKS_API_KEY" \
  -H "Idempotency-Key: order-42-reminder" \
  -d '{"to":"+972...","type":"text","text":"hi"}'
```

## Picking the sending number

If your workspace has **exactly one** connected WhatsApp session,
messages are sent from it automatically. With two or more active
sessions, include `from`:

```json
{ "to": "+1...", "type": "text", "text": "hi", "from": "+15551234567" }
```

Return codes: `409 no_active_session`, `409 ambiguous_session`,
`422 from_not_linked`.

## Status lifecycle

```
scheduled → queued → sending → delivered
                             ↘ failed
```

Poll the current state with `GET /v1/messages/{id}`. Read receipts
(`read_at` populated, `message.read` webhook fires) are delivered via
webhooks — see the [Webhooks guide](/docs/webhooks).

## Not yet supported

These message types are on the roadmap but not yet exposed in v1.0.
Use [the MCP server](/docs/mcp) or watch the [changelog](/docs/changelog) for ship dates:

- **Location** — share a coordinate with optional name/address.
- **Contacts (vcard)** — share a contact card.
- **Buttons / list** — interactive replies and selectable options.
