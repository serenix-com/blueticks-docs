This guide assumes you have an API key ([Authentication](/docs/authentication))
and an SDK client set up ([Quickstart](/docs/quickstart)).

There are two ways to send a WhatsApp message. Pick based on whether you want a
tracked record:

| Endpoint | When to use it | Tracked? |
| --- | --- | --- |
| [`POST /v1/scheduled-messages`](/docs/api/scheduled-messages/schedule-message) | **The default for almost everything.** Send right now *or* schedule for later. | **Yes** — returns a record you can list, poll for status, and edit. |
| [`POST /v1/messages/{chatId}`](/docs/api/messages/send-message) | A lightweight, fire-and-forget send into one chat. | No — dispatched directly, no record kept. |

<Callout type="info">
Despite the name, **`POST /v1/scheduled-messages` is the main "send a message"
endpoint** — not just for scheduling. Omit `sendAt` and the message goes out
immediately; add `sendAt` and it's scheduled for later. This guide uses it for
every example below.
</Callout>

The body is a flat object — set `type` to one of `text`, `media`, or `poll`;
`type` selects which fields are required. The optional fields `sendAt`
(schedule) and `replyTo` (quote a message) work on every variant.

## Send to a group or channel

The recipient is the `{chatId}` path segment — a phone number in international format OR a WhatsApp JID. JIDs come in three flavors:

- `<digits>@c.us` — individual chat (digits-only equivalent of an international-format phone number with `+` stripped).
- `<id>@g.us` — group chat (e.g. `120363427920657250@g.us`). Find a group's id with `GET /v1/groups` or by inspecting the URL in WhatsApp Web.
- `<id>@newsletter` — channel (e.g. `12345@newsletter`).

The send call is identical — just swap the recipient:

<Tabs groupId="sdk" persist items={['Python', 'Node.js', 'PHP', 'Ruby', 'Go', 'cURL']}>

```python
msg = bt.scheduled_messages.create(
    "120363427920657250@g.us",
    type="text",
    text="Team standup in 10 minutes!",
)

msg = bt.scheduled_messages.create(

    type="text",
    text="New announcement for all subscribers.",

```

## Send media

`type: "media"` takes the file as `mediaUrl` (an HTTPS URL) or `mediaBase64`
(the raw bytes base64-encoded, or a `data:` URL); `mediaUrl` wins if both are
present. The optional `mediaKind`
(`image` · `video` · `audio` · `document` · `sticker` · `voice` · `gif`)
is auto-detected from the URL or the `Content-Type` of the fetched asset
when omitted. `text` doubles as the caption under image/video in WhatsApp;
`mediaFilename` is shown for documents.

<Tabs groupId="sdk" persist items={['Python', 'Node.js', 'PHP', 'Ruby', 'Go', 'cURL']}>
  <Tab value="Python">

```python
import blueticks
bt = blueticks.Blueticks(api_key="BLUETICKS_API_KEY")

msg = bt.scheduled_messages.create(
    "+972501234567",
    type="media",
    media={
        "url": "https://cdn.example.com/receipt.pdf",
        "kind": "document",
        "filename": "receipt-A42.pdf",
        "caption": "Order #A-42",
    },
)
print(msg.id, msg.status)
```

  </Tab>

Voice notes are media with `mediaKind: "voice"`. The recipient is the `{chatId}`
path segment (e.g. `POST /v1/scheduled-messages/+972...`), so the body carries no `to`:

```json
{ "type": "media", "mediaUrl": "https://.../note.ogg", "mediaKind": "voice" }
```

`mediaUrl` must be `https://` and resolve to a non-private host — we fetch
and forward it on your behalf. If the file has no public URL, send the bytes
inline as `mediaBase64` instead.

## Schedule for later

`sendAt` (ISO 8601 with offset, ≥ 10 s in the future, ≤ 365 d out)
works on every type. The response status is `pending` — the message waits
until the `sendAt` time before dispatching. The recipient stays in the path
(`POST /v1/scheduled-messages/+972...`); only the body is shown here:

```json
{ "type": "text", "text": "Reminder", "sendAt": "2026-12-01T09:00:00Z" }
```

## Quote-reply to a prior message

Set `replyTo` to the serialized wire key of a message in the same chat
(returned as `data.waMessageKey._serialized` once the original has been sent).
Works with `text`, `media`, and `poll`. The recipient is the `{chatId}` path
segment (`POST /v1/scheduled-messages/+972...`), so it isn't in the body:

```json
{ "type": "text", "text": "Got it 👍", "replyTo": "false_972...@c.us_3EB0..." }
```

## React to a message

Reactions are a separate endpoint — `POST /v1/messages/reactions/{waMessageKey}`
with body `{ "emoji": "❤️" }`. `waMessageKey` is the message's complete
serialized wire key (`data.waMessageKey._serialized` from a send response). Pass an empty string to remove a
reaction. The optional `?chatId=` query param just narrows the engine lookup.

## Idempotency

Safe retries via the `Idempotency-Key` header. Same key + same body
within 24 h returns the original response (as a `200` replay instead of
`201`). Different body with the same key returns `409`.

## Status lifecycle

```
pending ──▶ confirmed ──▶ received ──▶ read ──▶ played
                     └──▶ failed
```

- **`pending`** — accepted by the API; waiting to dispatch (either immediately or at `sendAt`).
- **`confirmed`** — WhatsApp accepted the message (wire key received). `confirmedAt` is populated.
- **`received`** — double grey tick: the recipient's device has it. `receivedAt` is populated.
- **`read`** — double blue tick: the recipient opened the message. `readAt` is populated.
- **`played`** — voice note played by the recipient. `playedAt` is populated.
- **`failed`** — terminal. See the `failureReason` field on the message for the cause.

Poll the current state with `GET /v1/scheduled-messages/{id}`. Read receipts
(`readAt` populated, `message.read` webhook fires) are delivered via
webhooks — see the [Webhooks guide](/docs/webhooks).

## Immediate send into an open chat

The second send path from the top of this guide. `POST /v1/messages/{chatId}`
posts straight into one conversation and returns the WhatsApp wire `key`. It's
fire-and-forget — no record is created, so there's nothing to list, poll, or
edit afterwards. Reach for it only when you don't need tracking; otherwise
use `POST /v1/scheduled-messages` (above). The body is the same flat `text` /
`media` / `poll` shape, minus `to` (it's in the URL) and `sendAt`. It also
accepts `multipart/form-data` with the file in a `mediaFile` part, for media
sends without a public URL:

<ApiExample op="POST /v1/messages/{chatId}" kind="request" />

### Response

A successful send returns `201 Created`; the WhatsApp wire key is the object
under `data.waMessageKey` (its `_serialized` form is mirrored on `data.id`):

```json
{
  "success": true,
  "data": {
    "id": "true_972501234567@c.us_3EB0ABCDEF",
    "waMessageKey": {
      "fromMe": true,
      "remote": "972501234567@c.us",
      "id": "3EB0ABCDEF",
      "_serialized": "true_972501234567@c.us_3EB0ABCDEF"
    },
    "to": "972501234567@c.us",
    "type": "text",
    "text": "Got it 👍",
    "mediaUrl": null,
    "mediaKind": null,
    "pollQuestion": null,
    "status": "confirmed",
    "sendAt": null,
    "createdAt": "2026-06-12T09:00:00.000Z",
    "confirmedAt": "2026-06-12T09:00:01.000Z",
    "receivedAt": null,
    "readAt": null,
    "playedAt": null,
    "failedAt": null,
    "failureReason": null
  }
}
```

- **Channels** — WhatsApp channels are exposed as newsletters:

## Not yet supported

These message types are on the roadmap but not yet exposed in the v1 API.
Use [the MCP server](/docs/mcp) or watch the [changelog](/docs/changelog) for ship dates:

- **Location** — share a coordinate with optional name/address.
- **Contacts (vcard)** — share a contact card.
- **Buttons / list** — interactive replies and selectable options.
