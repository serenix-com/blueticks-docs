Every send goes through `POST /v1/scheduled-messages`. The body is a discriminated
union — set `type` to one of `text`, `media`, or `poll`. Optional fields
`send_at`, `from`, and `reply_to` work on every variant.

## Send a text message

<Tabs groupId="sdk" persist items={['Python', 'Node.js', 'PHP', 'cURL']}>
  <Tab value="Python">

```python
import blueticks
bt = blueticks.Blueticks(api_key="BLUETICKS_API_KEY")

msg = bt.scheduled_messages.create(
    to="+972501234567",
    type="text",
    text="Your order is on the way 🎉",
)
print(msg.id, msg.status)
```

  </Tab>
  <Tab value="Node.js">

```ts
import { Blueticks } from 'blueticks';
const bt = new Blueticks({ apiKey: 'BLUETICKS_API_KEY' });

const msg = await bt.scheduledMessages.create({
  to: '+972501234567',
  type: 'text',
  text: 'Your order is on the way 🎉',
});
```

  </Tab>
  <Tab value="PHP">

```php

$msg = $bt->scheduled_messages->create([
    'to'   => '+972501234567',
```

  </Tab>

## Send to a group or channel

The `to` field accepts an E.164 phone number OR a WhatsApp JID. JIDs come in three flavors:

- `<digits>@c.us` — individual chat (digits-only equivalent of an E.164 phone with `+` stripped).
- `<id>@g.us` — group chat (e.g. `120363427920657250@g.us`). Find a group's id with `GET /v1/groups` or by inspecting the URL in WhatsApp Web.
- `<id>@newsletter` — channel (e.g. `12345@newsletter`).

The send call is identical — just swap the `to`:

<Tabs groupId="sdk" persist items={['Python', 'Node.js', 'PHP', 'cURL']}>
  <Tab value="Python">

```python
import blueticks
bt = blueticks.Blueticks(api_key="BLUETICKS_API_KEY")

# Send to a group
msg = bt.scheduled_messages.create(
    to="120363427920657250@g.us",
    type="text",
    text="Team standup in 10 minutes!",
)

# Send to a channel
msg = bt.scheduled_messages.create(
    to="12345@newsletter",
    type="text",
    text="New announcement for all subscribers.",
)
```

  </Tab>
  <Tab value="Node.js">


## Send media

`type: "media"` accepts an HTTPS URL plus an optional `kind`
(`image` · `video` · `audio` · `document` · `sticker` · `voice` · `gif`).
When `kind` is omitted, the engine auto-detects from the URL or the
`Content-Type` of the fetched asset. `caption` shows under image/video
in WhatsApp; `filename` is shown for documents.

<Tabs groupId="sdk" persist items={['Python', 'Node.js', 'PHP', 'cURL']}>
  <Tab value="Python">

```python
import blueticks
bt = blueticks.Blueticks(api_key="BLUETICKS_API_KEY")

msg = bt.scheduled_messages.create(
    to="+972501234567",
    type="media",
    media={
        "url": "https://cdn.example.com/receipt.pdf",
        "kind": "document",
        "filename": "receipt-A42.pdf",
        "caption": "Order #A-42",
    },
)
print(msg.id, msg.status)
```

  </Tab>
  <Tab value="Node.js">


Voice notes are media with `kind: "voice"`:

```json
{ "to": "+972...", "type": "media", "media": { "url": "https://.../note.ogg", "kind": "voice" } }
```

`media.url` must be `https://` and resolve to a non-private host. We
fetch and forward it on your behalf.

## Send a poll

`options` must contain 2–12 entries. `allow_multiple` lets recipients
pick more than one (default `false`).

<Tabs groupId="sdk" persist items={['Python', 'Node.js', 'PHP', 'cURL']}>
  <Tab value="Python">

```python
import blueticks
bt = blueticks.Blueticks(api_key="BLUETICKS_API_KEY")

msg = bt.scheduled_messages.create(
    to="+972501234567",
    type="poll",
    poll={
        "question": "Pizza tonight?",
        "options": ["Yes", "No", "Maybe"],
        "allow_multiple": False,
    },
)
print(msg.id, msg.status)
```

## Schedule for later

`send_at` (ISO 8601 with offset, ≥ 10 s in the future, ≤ 365 d out)
works on every type. The response status is `scheduled` instead of
`queued`.

```json
{ "to": "+972...", "type": "text", "text": "Reminder", "send_at": "2026-12-01T09:00:00Z" }
```

## Quote-reply to a prior message

Set `reply_to` to the wire `key` of a message in the same chat
(returned in `MessageResponse.key` once the original has been sent).
Works with `text`, `media`, and `poll`.

```json
{ "to": "+972...", "type": "text", "text": "Got it 👍", "reply_to": "false_972...@c.us_3EB0..." }
```

## React to a message

Reactions are a separate endpoint — `POST /v1/chats/{chat_id}/messages/{key}/reactions`
with body `{ "emoji": "❤️" }`. Pass an empty string to remove a reaction.
Use the `key` from `MessageResponse`, not the `id`.

<Tabs groupId="sdk" persist items={['Python', 'Node.js', 'PHP', 'cURL']}>
  <Tab value="Python">

```python
import blueticks
bt = blueticks.Blueticks(api_key="BLUETICKS_API_KEY")

bt.chats.react(
    chat_id="972501234567@c.us",
    key="false_972501234567@c.us_3EB0ABCDEF",
    emoji="❤️",
)
```

## Idempotency

Safe retries via the `Idempotency-Key` header. Same key + same body
within 24 h returns the original response. Different body with the
same key returns `409 idempotency_key_conflict`.

## Picking the sending number

If your workspace has **exactly one** connected WhatsApp session,
messages are sent from it automatically. With two or more active
sessions, include `from`:

```json
{ "to": "+1...", "type": "text", "text": "hi", "from": "+15551234567" }
```

Return codes: `409 no_active_session`, `409 ambiguous_session`,
`422 from_not_linked`.

## Status lifecycle

```
scheduled ──▶ queued ──▶ sending ──┬──▶ delivered ──▶ read
                                   └──▶ failed
```

- **`scheduled`** — set when `send_at` is in the future; transitions to `queued` at the scheduled time.
- **`queued`** — accepted by the API, waiting for an active WhatsApp session to dispatch.
- **`sending`** — dispatched to the engine; on the wire.
- **`delivered`** — the recipient's WhatsApp acknowledged delivery. May further transition to `read`.
- **`read`** — terminal. The recipient opened the message (read receipt received).
- **`failed`** — terminal. See the `failure_reason` field on the message for the cause.

Poll the current state with `GET /v1/scheduled-messages/{id}`. Read receipts
(`read_at` populated, `message.read` webhook fires) are delivered via
webhooks — see the [Webhooks guide](/docs/webhooks).

## Not yet supported

These message types are on the roadmap but not yet exposed in v1.0.
Use [the MCP server](/docs/mcp) or watch the [changelog](/docs/changelog) for ship dates:

- **Location** — share a coordinate with optional name/address.
- **Contacts (vcard)** — share a contact card.
- **Buttons / list** — interactive replies and selectable options.
