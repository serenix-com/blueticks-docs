# Comprehensive Doc ⇄ OpenAPI Validation — Design

**Date:** 2026-06-06
**Repo:** `serenix-com/blueticks-docs`
**Status:** Approved design — ready for implementation plan.

## Goal

Docs examples can never silently diverge from `openapi.json`, enforced in CI. Every code block in the guides is either **generated from the spec** (drift impossible) or **validated against the spec** (drift caught). Prose/text is out of scope — only code blocks and operation coverage are checked.

## Background — what exists today

The merged `validate:examples` pipeline (`scripts/validate-examples.ts` + `scripts/lib/**`) already:

- Scans **top-level guides only** (`content/docs/*.mdx`); excludes the generated `content/docs/api/**`.
- Validates **request bodies** in code examples (cURL/JSON/Python/Node/TS) against request schemas: `unknown-field`, `bad-enum`, `missing-required`, `dead-endpoint`, `unparseable`.
- Resolves each `<Tabs>` group → an OpenAPI operation (cURL URL anchors it; else SDK-method→path via `scripts/lib/sdk-mapping.ts`).
- Has an autofix for single-token renames/enum fixes (`validate:examples:fix`).
- Is gated in CI (`.github/workflows/ci.yml`) alongside a "generated `content/docs/api/` is in sync" check.

The reference pages under `content/docs/api/**` are generated by **`fumadocs-openapi`** (`scripts/generate-openapi-pages.ts` → `generateFiles()` emitting `<APIPage>` tags). The per-language request/response code samples on those pages come from fumadocs-openapi's own sample engine — not hand-rolled.

## Architecture — one source, two surfaces

```
                    openapi.json  ──────────────┐
                          │                      │
          fumadocs-openapi sample engine         │
        (same path <APIPage> already uses;       │
         wrapped by lib/example-engine.ts)        │
                    │            │               │
        ┌───────────┘            └──────────┐    │
        ▼                                   ▼    ▼
  GENERATE                             VALIDATE (CI gate)
  <ApiExample op kind status lang/>    scan every guide MDX code block
  renders generated tabs at build      → request/response/params checked
  → correct by construction;           → coverage: every /v1 op used ≥1×
    validator skips it                 → hard-gate; suppress via directive
```

The two halves share the understanding of "the canonical example for operation X" — generation **emits** it, validation **compares against** it. As guides migrate to `<ApiExample>`, the validator's raw-block surface shrinks toward just the coverage gate, which always stays on.

## Component A — `lib/example-engine.ts` (shared core)

A single module that, given `openapi.json` + an operation key, produces the canonical example artifacts both surfaces need.

**Interface:**

```ts
type OpKey = string; // "POST /v1/messages" — METHOD + spec path
type Lang = 'curl' | 'python' | 'node' | 'json';

interface RequestExample {
  method: string;
  url: string;                  // server URL + path with path-params filled
  headers: Record<string, string>;   // auth + required header params
  query: Record<string, string>;     // required query params
  body: unknown | null;         // JSON request body, or null if none
  perLang: Record<Lang, string>;      // rendered code per language
}
interface ResponseExample {
  status: string;               // e.g. "200"
  body: unknown;                // JSON response body for that status
  perLang: Record<'json', string>;
}

function resolveOp(spec, opKey): ResolvedOperation | null;
function buildRequestExample(spec, opKey): RequestExample | null;
function buildResponseExample(spec, opKey, status?): ResponseExample | null; // status default = first 2xx
```

**Value sourcing (in priority order):** field-level `example` → `examples` → `default` from the spec; else synthesize from `type`/`format`/`enum` (e.g. `format:email → user@example.com`, `format:date-time → 2026-01-01T00:00:00Z`, `enum → enum[0]`, `string → "string"`, `integer → 0`). Discriminated unions pick the first `oneOf` variant (honoring the discriminator mapping).

**Reuse over reinvention:** if `fumadocs-openapi` exposes a per-operation sample API, the engine wraps it so guide output is byte-identical to `<APIPage>`. If not, the engine renders the same `<Tabs>`+`CodeBlock` shape itself. Either way this module is the *single* definition of "the canonical example," consumed by both the component and the validator's expected-form check.

## Component B — `<ApiExample>` MDX component (generate surface)

```
<ApiExample
  op="POST /v1/messages"   // required — the operation key
  kind="request"           // "request" | "response"; default "request"
  status="200"             // response only; default first 2xx
  lang="curl"              // optional; omit → all languages as <Tabs>
/>
```

- At build, resolves `op` via `lib/example-engine.ts` and renders the generated request (URL + auth + path/query params + body) or response body, in the requested language(s), using the same `<Tabs>`/`CodeBlock` markup as the reference pages.
- Registered once in `components/mdx.tsx` (next to `<APIPage>`), usable in any guide MDX.
- An unresolvable `op` (not in spec) is a **build error** — a guide referencing a dead operation fails the build, not just the validator.
- Each `<ApiExample op=...>` use counts as a guide reference to that op, feeding the coverage gate.

## Component C — validator extensions (validate surface)

Extend the existing pipeline; do not fork it.

**Scope:** widen the file glob from `content/docs/*.mdx` to `content/docs/**/*.mdx`, still excluding `content/docs/api/**` (generated `<APIPage>`, no raw blocks). `<ApiExample>` invocations are JSX, not fenced code, so they're naturally skipped.

**New checks:**

1. **Response bodies** — classify a code block as a response candidate when it carries an explicit `{/* example:response 200 */}` marker, **or** it is a JSON block under a "Response" heading immediately following a resolved request group. Ambiguous JSON with neither signal → require the marker (no silent guessing). Validate the block's fields/enums/shape against the resolved op's response schema for that status. New finding kind: `response-shape`.
2. **Params must exist** — every path/query/header parameter used in an example must be declared on that operation in the spec. New finding kind: `unknown-param`.
3. **Coverage gate** — every `/v1` operation in `openapi.json` must be referenced by ≥1 guide: an `<ApiExample op=...>` **or** a hand-written code block that resolves to it. Operations referenced by neither fail. New finding kind: `coverage-gap`.

Existing request-body checks (`unknown-field`, `bad-enum`, `missing-required`, `dead-endpoint`) are unchanged.

**Hard-gate + suppression:**

- Every finding fails CI (non-zero exit).
- **Inline:** `{/* validate:ignore <kind> — reason */}` directly above a block or `<Tabs>` group suppresses that finding kind for it. The reason text is required.
- **File-level:** `.validateignore` at the docs root exempts operations from the **coverage** gate only (e.g. internal ops with no guide). One op-key glob per line, each with a trailing `# reason`.
- Non-API code blocks (`npm install`, shell snippets) auto-skip when they don't resolve to an operation — no marker needed. The existing `{/* example:skip reason */}` annotation still works for explicit skips.

**Finding model:** the existing `Finding` interface (`scripts/lib/types.ts`) gains kinds `response-shape`, `unknown-param`, `coverage-gap`. Reporting stays as-is: grouped by file with line numbers, suggestions where high-confidence, non-zero exit on any error.

## CI

No new job. The existing `validate:examples` step in `.github/workflows/ci.yml` picks up the new checks automatically. The generated-content-in-sync check stays. A guide that references a dead op now also fails earlier, at `pnpm build`, via `<ApiExample>`'s build-time resolution.

## Testing

- **Engine** (`lib/__tests__/example-engine.test.ts`): value sourcing precedence (example→examples→default→synthesized); path/query/header assembly; discriminated-union variant selection; request vs response per status; unresolvable op → null.
- **Component**: renders request tabs for a known op; renders response body for a given status; unknown op → throws at build.
- **Validator** (extend `scripts/lib/__tests__/**`): a drifted response field → `response-shape`; an undeclared query param → `unknown-param`; an op with no guide reference → `coverage-gap`; `<ApiExample>` satisfies coverage; `{/* validate:ignore */}` suppresses; `.validateignore` exempts a coverage op; non-API block auto-skips.

## Phasing (for the implementation plan)

1. `lib/example-engine.ts` + tests (shared core first).
2. `<ApiExample>` component + mdx wiring + tests.
3. Validator extensions (response, params, coverage) + finding kinds + suppression + tests.
4. Migrate a first guide section to `<ApiExample>` as a worked example; turn the gate on.

## Out of scope

- Prose/text claim validation (explicitly dropped — code blocks only).
- Live execution of examples against a running API.
- Enriching backend schemas with per-field `.openapi({ example })` — improves generated-value quality; a follow-on, not required for this to ship.
- The `content/docs/api/**` reference pages themselves (already generated + sync-checked).
